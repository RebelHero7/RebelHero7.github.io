AbstractQueuedSynchronizer源码解析（上）

AbstractQueuedSynchronizer(以下简称AQS)应该都知道，是很多并发类的基石，ReentrantLock，CountDownLatch等都是基于它。

先来看它的变量，比较简单，就三个。

/**
 * 头部
 */
private transient volatile Node head;

/**
 * 等待队列尾部
 */
private transient volatile Node tail;

/** 
 * 当前锁的状态，0：没有被占用， > 0: 有线程占用了，由于是可重入锁，每次重入都+1，所以会 > 0
 */
private volatile int state;

可以看见，有头有尾的，属性是Node，也就是说，等待队列中的每个线程被封装成了Node，这个Node是AQS的内部静态类。

static final class Node {
    /** 标识节点为分享模式 */
    static final Node SHARED = new Node();
    /** 标识节点当前在独占模式 */
    static final Node EXCLUSIVE = null;

    /** 该节点已取消等待 */
    static final int CANCELLED =  1;
    /** next 节点需要被唤醒 */
    static final int SIGNAL    = -1;
    /** 表示正在等待条件 */
    static final int CONDITION = -2;
    /**
     * 表示下一个acquireShared应该无条件传播
     */
    static final int PROPAGATE = -3;

    // 等待状态
    volatile int waitStatus;
    // 前节点
    volatile Node prev;
    // 后节点
    volatile Node next;

    /**
     * 包裹的线程
     */
    volatile Thread thread;

    /**
     * 链接到等待条件的下一个节点，或者共享特殊值
     */
    Node nextWaiter;

    /**
     * 节点是否在共享模式下等待
     */
    final boolean isShared() {
        return nextWaiter == SHARED;
    }

    /**
     * 获取上一个节点
     */
    final Node predecessor() throws NullPointerException {
        Node p = prev;
        if (p == null)
            throw new NullPointerException();
        else
            return p;
    }
    // 省略构造方法。
}

根据prev 和 next两个属性可以判断，Node就是一个双向链表。其他的属性我已经写在注释里了，就不再重复了。

加锁源码

下面以最常见的ReentrantLock来分析一下AQS中一些方法调用。

最开始，我们从ReentrantLock的lock()方法进入。

public void lock() {
    sync.lock();
}

可以看见，是封装了sync的lock的，而Sync是一个在ReentrantLock的抽象静态内部类，其下还有NonfairSync和FairSync两个类，就是公平锁和非公平锁。新建ReentrantLock可以指定使用公平锁还是非公平锁，默认是非公平的，下面我们直接以非公平锁举例。

final void lock() {
    // 使用CAS替换。
    if (compareAndSetState(0, 1))
        setExclusiveOwnerThread(Thread.currentThread());
    else
        acquire(1);
}

上面也说了，如果state是0表示没有线程持有锁，使用CAS看看能不能占用锁，能，就将当前持有线程改为自己，不能，就进入acquire方法。

public final void acquire(int arg) {
    // 再次尝试去获取锁，没成功就去排队咯，acquireQueued这个方法会返回中断状态，如果为true就中断。
    if (!tryAcquire(arg) &&
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();
}

下面看一下tryAcquire

protected final boolean tryAcquire(int acquires) {
    return nonfairTryAcquire(acquires);
}

final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    // 如果状态为0，说明没线程持有。
    if (c == 0) {
        // 再次尝试，成功就说明占有锁了。
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    // 说明已经有线程持有锁了，判断是不是这个线程是不是自己，是自己就重入给state + 1
    else if (current == getExclusiveOwnerThread()) {
        int nextc = c + acquires;
        if (nextc < 0)
            throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }
    // 到这说明尝试获取锁失败。
    return false;
}

因为多线程瞬息万变，可能在执行到tryAcquire的时候，之前持有锁的已经解锁了，因此再次尝试，当尝试失败的时候，要么说明之前持有的还在持有，或者被其他线程抢先了，总之，失败就进入到acquireQueued排队去吧，在排队前先调用addWaiter将线程封装成Node再排队。

private Node addWaiter(Node mode) {
    // 将当前线程封装成Node。
    Node node = new Node(Thread.currentThread(), mode);
    // 获取这个等待队列的尾节点。
    Node pred = tail;
   
    if (pred != null) {
        // 由于是双向的，因此，将当前线程的前节点设为之前的尾节点。
        node.prev = pred;
        // 使用CAS线程设为尾节点，成功的话，将之前尾节点的next节点设为当前线程并且返回。
        if (compareAndSetTail(pred, node)) {
            pred.next = node;
            return node;
        }
    }
    // 1. 尾节点为空，该线程是第一个排队的。
    // 2. 尾节点不为空， 但上面CAS失败。
    enq(node);
    return node;
}

private Node enq(final Node node) {
    for (;;) {
        Node t = tail;
        // 上面说的情况1
        if (t == null) { 
            // 新建一个头节点，并使用CAS设置。
            if (compareAndSetHead(new Node()))
                // 成功，就将尾部指向头结点
                // 注意，此时还没有跳出循环！！
                tail = head;
        } else {
            // 还是重复上面的逻辑，一直循环到当前线程排到尾节点。
            node.prev = t;
            if (compareAndSetTail(t, node)) {
                t.next = node;
                // 再次注意，返回的之前的尾节点！！
                // 不过addWaiter也没用到这个返回值，但是后面分析其他方法的时候会用到，现在这里提个醒。
                return t;
            }
        }
    }
}

此处有一点需要指出来，enq方法中，尾节点为空时，说明等待队列为空，此时使用了new Node()新增一个头节点，尾节点也指向它，再将当前线程设置为尾节点的next节点，而不是将当前线程设置为头节点！也就是说，第一次真正的等待队列其实是从头节点的next节点开始的！head最开始的时候只是一个不包含任何线程的Node下面继续分析，此时已经将当前节点封装成Node并排队到最后了，下面看看acquireQueued。

final boolean acquireQueued(final Node node, int arg) {
    // 失败标志。
    boolean failed = true;
    try {
        // 中断标志。
        boolean interrupted = false;
        for (;;) {
            // 获取当前节点的上一个节点。
            final Node p = node.predecessor();
            // 如果上一个节点是head，说明当前是排队的第一个。
            // 可能上个持有锁的已经解锁了，那么再次试着获取持有。
            if (p == head && tryAcquire(arg)) {
                // 将当前节点设置为头节点。
                setHead(node);
                p.next = null; 
                failed = false;
                return interrupted;
            }
            // 这个等下直接进去看。
            if (shouldParkAfterFailedAcquire(p, node) &&
                parkAndCheckInterrupt())
                interrupted = true;
        }
    } finally {
        // 失败了，取消正在进行的获取尝试。
        if (failed)
            cancelAcquire(node);
    }
}

此时又有一个重点了，可以发现，当节点已经获取到锁的使用权的时候，会将当前节点设置为头节点，也就是说，你可以将head节点看成持有锁的线程，为什么说可以，后面会讲清楚。再次重申， 真正的等待队列其实是从头节点的next节点开始的！ 可以的意思是，因为开始head是个不包含线程的Node。

下面继续分析

private static boolean shouldParkAfterFailedAcquire(Node pred, Node node) {
    // 看上面的调用，node是当前节点， pred是当前节点的前一个节点。
    int ws = pred.waitStatus;
    // 前一个节点的waitStatus，为SIGNAL，也就是-1时，直接返回true。
    if (ws == Node.SIGNAL)
        return true;
    // 如果 > 0，就是CANCELLED， 说明前一个节点已经取消等待了！
    if (ws > 0) {
        do {
            // 一直向前找，找到waitStatus <= 0 的节点，也就是还在等待的节点。
            node.prev = pred = pred.prev;
        } while (pred.waitStatus > 0);
        pred.next = node;
    } else {
        /*
         * 说明waitStatus <= 0 && != -1, 那么将前节点的waitStatus设置为SIGNAL，即-1
         */
        compareAndSetWaitStatus(pred, ws, Node.SIGNAL);
    }
    return false;
}

// 将线程阻塞。
private final boolean parkAndCheckInterrupt() {
    LockSupport.park(this);
    return Thread.interrupted();
}

这里还有一个重点。

首先，每个节点初始化的时候，waitStatus其实都是0，排到队尾，那也是0。

其次，parkAndCheckInterrupt进行真正的阻塞前，还会执行shouldParkAfterFailedAcquire方法。方法逻辑为：获取上个节点的waitStatus状态，

  • 如果其状态为-1，return true
  • 如果 > 0，说明上个节点已经取消等待了，一直找，找到没有取消的节点为止，并排到其后面，return false
  • 如果<= 0 && != -1，那么将上个节点的waitStatus置为SIGNAL，return false。

可以看到后面两个都会返回false，那么是不会执行parkAndCheckInterrupt的，会再次循环，直到将上个节点的waitStatus置为SIGNAL。

也就是说，线程进入真正的阻塞前，是要将上个节点的状态置为SIGNAL的，这个SIGNAL，就是说，上个节点解锁的时候需要唤醒下个正在等待的线程，这也就解释了我之前在SIGNAL上的注释。

下面我们来看一下 cancelAcquire方法

private void cancelAcquire(Node node) {
    if (node == null)
        return;

    node.thread = null;

    // 获取上个节点。
    Node pred = node.prev;
    // 说明上个节点也取消等待了，那么一直往前找，找到没取消的，假设叫A。
    while (pred.waitStatus > 0)
        node.prev = pred = pred.prev;

    // 获取A的下个节点
    Node predNext = pred.next;

    node.waitStatus = Node.CANCELLED;

    // 如果当前要取消的节点是尾节点，那么重新设置尾节点。
    if (node == tail && compareAndSetTail(node, pred)) {
        compareAndSetNext(pred, predNext, null);
    } else {
        // 说明不是尾节点，或者，上面设置尾节点失败。
        int ws;
        // 如果上个节点不为头节点 &&
        // 上个节点的waitStatus 为SIGNAL || ws <= 0 的话，将其改为SIGNAL，总之就是保证上个节点的waitStatus 为SIGNAL,确保它能唤醒下个节点。
        // && pred的线程不为null
        if (pred != head &&
            ((ws = pred.waitStatus) == Node.SIGNAL ||
             (ws <= 0 && compareAndSetWaitStatus(pred, ws, Node.SIGNAL))) &&
            pred.thread != null) {
            Node next = node.next;
            if (next != null && next.waitStatus <= 0)
                compareAndSetNext(pred, predNext, next);
        } else {
            // 上面条件不符合就进入这里。
            unparkSuccessor(node);
        }
        node.next = node; // help GC
    }
}

// 唤醒后继节点。
private void unparkSuccessor(Node node) {
    
    int ws = node.waitStatus;
    
    if (ws < 0)
        compareAndSetWaitStatus(node, ws, 0);

    // 获取后继节点
    Node s = node.next;
    // 如果为空，或者，其状态为取消，那么一直往后找，直到找到还在等待的。
    if (s == null || s.waitStatus > 0) {
        s = null;
        for (Node t = tail; t != null && t != node; t = t.prev)
            if (t.waitStatus <= 0)
                s = t;
    }
    // 唤醒下个还在等待的线程。
    if (s != null)
        LockSupport.unpark(s.thread);
}

这个没什么重点，慢慢看，逻辑简单明了。

解锁

下面直接看解锁并如何唤醒下个等待队列的线程的，使用ReentrantLock.unlock()来解锁

public void unlock() {
    sync.release(1);
}

public final boolean release(int arg) {
    if (tryRelease(arg)) {
        Node h = head;
        if (h != null && h.waitStatus != 0)
            unparkSuccessor(h);
        return true;
    }
    return false;
}

protected final boolean tryRelease(int releases) {
    int c = getState() - releases;
    // 你都不是持有锁线程的谈何解锁，抛异常！
    if (Thread.currentThread() != getExclusiveOwnerThread())
        throw new IllegalMonitorStateException();
    // 表示是否已经完全解锁了。
    boolean free = false;
   	// 说明已完全解锁。
    if (c == 0) {
        free = true;
        setExclusiveOwnerThread(null);
    }
    setState(c);
    return free;
}

可以看见，tryRelease中，如果状态是0了，说明已经锁已经没有线程持有了，将持有的线程置为null，并设置free为true。不过这个锁是不是完全解锁，都将状态更新。

如果tryRelease返回的是true，说明已经完全释放锁了，那么可以唤醒下个线程了，这唤醒下个线程的unparkSuccessor已经在上一节的最后分析过了。

实例分析

我们简单的过一下加锁逻辑，解锁比较简单就不过了。

  1. 假设只有A、B、C三个线程来lock，假设说，A首先获取到锁了，lock的时候，就不会执行到acquire这个方法。但是B和C会执行到。假如说这两个线程执行tryAcquire的时候，A还持有锁（如果已经没有持有了，B或C就有一个能持有），就会执行到acquireQueued(addWaiter(Node.EXCLUSIVE), arg)这个方法。
  2. 假如说B先执行到addWaiter方法，由于它是第一个需要阻塞的线程，tail == null，那么就会执行到enq这个方法，执行两次for循环，第一次new Node()（假设叫Z），并将head 和 tail都指向它，第二次循环tail != null ，那么将B线程为tail，B的prev也指向刚刚新建的 Z node， Z node 的next 也指向B 。 然后C进入了这个addWaiter，发现tail != null ，那么将它自己设为tail，此时的等待队列是 Z（head节点，没线程）-> B -> C。
    • 当然，理想化是上面描述的这样，现实可能是，B、C同时进入addWaiter，可能也会同时进入到enq，（这前面的两个同时不是真的同时！）然后都判断出tail为null，但还是必会有一个线程new 出 Z node 的并且CAS Head成功的，可能是C new出来并设置成功的话，B执行compareAndSetHead的时候就会失败，那么进行下一次循环，最后可能是Z（没线程）-> C -> B。
    • 注： 此时，三个节点的waitStatus都为0。
  3. B、C都会到acquireQueued方法，进入for循环，由于B的prev是 Z node也就是头节点，那么会再次去尝试获取锁。分析B的情况：
    • A线程已经解锁，不持有锁了！那么，B获取锁成功了，那么head就会设置为B！也就是说，此时的head节点为持有锁的节点！
    • A还没有解锁，tryAcquire失败，就会执行shouldParkAfterFailedAcquire，由于Z的waitStatus目前是0，因此CAS 设为SIGNAL，然后返回false，此时B还没有阻塞，将再次for循环，再次重复该1、2，但这次进入shouldParkAfterFailedAcquire由于Z的waitStatus是-1了，返回true，那么执行parkAndCheckInterrupt阻塞B，进行排队。
  4. C的情况就不说了，和上面B类似。

总结

其实重点啥的正文都已经说的很清楚了，但还是，最后简单的总结一下：

  • 等待队列中，head节点的意思虽然说是等待队列的head，但它自始至终都没等待过，它的值要么是空节点，要么是持有锁的线程。真正阻塞等待的是它的next节点！！

  • waitStatus这个状态，很重要，本文只涉及到了CANCELLED和SIGNAL，下篇应该能涉及到另外两个， 而SIGNAL是表示，当前节点解锁的时候需要唤醒下个节点，而且这个状态不是在放入到等待队列中设置的，而是在下个节点进入等待队列的时候设置为SIGNAL的。
