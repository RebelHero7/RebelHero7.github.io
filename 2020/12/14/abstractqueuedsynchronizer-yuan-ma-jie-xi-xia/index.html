AQS源码解析（下）

我们由一段生产者-消费者代码进入这篇文章

public class Buffer {
    private List<String> queue;
    private int size;

    private final Lock lock = new ReentrantLock();
    // 读线程条件
    private final Condition notEmpty = lock.newCondition();
    // 写线程条件
    private final Condition notFull = lock.newCondition();

    public Buffer() {
        this(10);
    }

    public Buffer(int size) {
        this.size = size;
        queue = new ArrayList<>();
    }

    // 生产
    public void put() throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == size) {
                notNull.await();
            }
            queue.add("kkkkk");
            
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    // 消费
    public void take() throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == 0) {
                notEmpty.await();
            }
            queue.remove(queue.size() - 1);
            notFull.signal();
        } finally {
            lock.unlock();
        }

    }
}

一段经典的生产者-消费者代码，就不解读细节了。condition 是依赖ReentrantLock的，调用 await 进入等待还是 signal 唤醒，都必须获取到锁才能进行操作。其实可以看出来， 这个是和Object.wait()，Object.notify() 或 Object.notifyAll()一样的，只是说condition 是基于ReentrantLock的。

下面看一下，Condition的变量，比较简单，就两个：

public class ConditionObject implements Condition, java.io.Serializable {
    private static final long serialVersionUID = 1173984872572414699L;
    /** 条件队列第一个 */
    private transient Node firstWaiter;
    /** 条件队列的最后一个 */
    private transient Node lastWaiter;

在这里我得补充一个重点：

  1. 这里的条件队列，并不同于上篇所讲的等待队列！两者不可混淆，Node类中，有一个nextWaiter是给条件队列用的，而没有一个prevWaiter，说明条件队列是一个单向链表！

  2. 当进入await方法，代表的是线程进入了条件队列中，而非等待队列，等signal时，不是说立刻就能拿到锁的！而是将条件队列中的节点放入到等待队列继续去排队！

await方法

不多说，直接看await方法，

public final void await() throws InterruptedException {
    // 如果线程已中断。
    if (Thread.interrupted())
        throw new InterruptedException();
    // 将当前线程包装成Node，并添加到条件队列中的尾部
    Node node = addConditionWaiter();
    // 释放拥有的锁，并返回之前的重入次数，也就是说，如果说重入了3次，那么返回3。
    int savedState = fullyRelease(node);
    int interruptMode = 0;
    // 如果不在等待队列中，那么阻塞。
    while (!isOnSyncQueue(node)) {
        LockSupport.park(this);
        // 唤醒后进行的操作，不谈这个了。
        if ((interruptMode = checkInterruptWhileWaiting(node)) != 0)
            break;
    }
    // 到这说明唤醒了，那么重新获取
    if (acquireQueued(node, savedState) && interruptMode != THROW_IE)
        interruptMode = REINTERRUPT;
    if (node.nextWaiter != null) // clean up if cancelled
        unlinkCancelledWaiters();
    if (interruptMode != 0)
        reportInterruptAfterWait(interruptMode);
}

一步一步来看，上面逻辑。

首先看下，如何将线程添加到条件队列，addConditionWaiter方法。

private Node addConditionWaiter() {
    // 获取条件队列的尾部。
    Node t = lastWaiter;
    // 如果不为空 && waitStatus不为CONDITION状态。
    if (t != null && t.waitStatus != Node.CONDITION) {
        // 清除状态不为CONDITION的节点。
        unlinkCancelledWaiters();
        // 重新获取。
        t = lastWaiter;
    }
    // 将当前线程包装成状态为CONDITION 的node。
    Node node = new Node(Thread.currentThread(), Node.CONDITION);
    // 插入到条件队列中。
    if (t == null)
        firstWaiter = node;
    else
        t.nextWaiter = node;
    lastWaiter = node;
    return node;
}

// 总体来说就是遍历清除状态不为CONDITION的节点， 是一个很简单的链表操作。
private void unlinkCancelledWaiters() {
    // 获取条件队列头部。
    Node t = firstWaiter;
    // 记录的是遍历时的上个节点，刚开始当然就是null了。
    Node trail = null;
    // 遍历，将已经取消等待的节点清除。
    while (t != null) {
        Node next = t.nextWaiter;
        // 如果节点状态不为CONDITION，就被清除。
        if (t.waitStatus != Node.CONDITION) {
            t.nextWaiter = null;
            if (trail == null)
                firstWaiter = next;
            else
                trail.nextWaiter = next;
            if (next == null)
                lastWaiter = trail;
        }
        else
            trail = t;
        t = next;
    }
}

逻辑很简单，就是将当前线程封装成Node， 并放至条件队列尾部。

下面继续看一下fullyRelease方法。

final int fullyRelease(Node node) {
    boolean failed = true;
    try {
        // 获取status，重入锁，可以 >= 1
        int savedState = getState();
        // 然后释放这些锁。
        if (release(savedState)) {
            failed = false;
            return savedState;
        } else {
            throw new IllegalMonitorStateException();
        }
    } finally {
        if (failed)
            node.waitStatus = Node.CANCELLED;
    }
}

这个也很简单，就不多说了。

下面看下while循环的条件isOnSyncQueue方法：

// 判断节点是否在等待队列中，注意，是等待队列，而不是条件队列！！
final boolean isOnSyncQueue(Node node) {
    // 如果说，等待状态为CONDITION，
    // 或者是，prev为null，因为prev是在等待队列中使用的，说明不在，返回false。
    if (node.waitStatus == Node.CONDITION || node.prev == null)
        return false;
    // 如果有后继节点，说明肯定在等待队列中。
    if (node.next != null) 
        return true;
    // 从尾部遍历找到该节点。
    // 到这里说明节点的prev != null， 但是为什么还有遍历循环一次。
    // 结合上一篇可以看到，当节点进入等待队列尾部时，会先设置node.prev = tail，再CAS设置节点为尾部，CAS可能会失败！
    return findNodeFromTail(node);
}

// 就，遍历然后判断。
private boolean findNodeFromTail(Node node) {
    Node t = tail;
    for (;;) {
        if (t == node)
            return true;
        if (t == null)
            return false;
        t = t.prev;
    }
}

我们现在从await方法抽出来，先来看看signal方法， 如何唤醒线程，再来看await继续的逻辑。

signal方法

public final void signal() {
    // 如果本身都不是拥有锁的线程，抛出异常！
    if (!isHeldExclusively())
        throw new IllegalMonitorStateException();
    // 获取条件队列的第一个
    Node first = firstWaiter;
    if (first != null)
        // 
        doSignal(first);
}

private void doSignal(Node first) {
    do {
        // 获取头节点的后继节点，为空，说明就一个，于是把尾节点也置为null。
        if ( (firstWaiter = first.nextWaiter) == null)
            lastWaiter = null;
        // 由于要去等待队列了，因此断开连接。
        first.nextWaiter = null;
        } while (!transferForSignal(first) &&
             (first = firstWaiter) != null);
}
final boolean transferForSignal(Node node) {
    /*
     * 由于要去等待队列了，因此把状态改为0。
     */
    if (!compareAndSetWaitStatus(node, Node.CONDITION, 0))
        return false;

    /*
     * 进入等待队列，请注意，这里返回的是上个节点，而不是该节点！关于这一点我上篇文章已经说了！
     */
    Node p = enq(node);
    int ws = p.waitStatus;
	  // 如果上个节点的waitStatus >0 说明取消等待，说明可以唤醒
    // 上篇也说过，插入之后要将上个节点的waitStatus 设SIGNAL。
    if (ws > 0 || !compareAndSetWaitStatus(p, ws, Node.SIGNAL))
        LockSupport.unpark(node.thread);
    return true;
}

下面看看acquireQueued

// 第二个参数，就是阻塞之前，重入的次数！我解除阻塞了肯定也必须重入相应的数量才是~
// 这个方法上一篇已经解释过了，就不再复述，感兴趣的可以去上一篇看一下。
final boolean acquireQueued(final Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            final Node p = node.predecessor();
            if (p == head && tryAcquire(arg)) {
                setHead(node);
                p.next = null; // help GC
                failed = false;
                return interrupted;
            }
            if (shouldParkAfterFailedAcquire(p, node) &&
                parkAndCheckInterrupt())
                interrupted = true;
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}
